# Local AI Model Hosting Project Plan

## 1. Project Overview

This project builds a locally hosted AI chat system on a Lenovo Legion 5 Pro 2023 R9000P with an NVIDIA GeForce RTX 4060 Laptop GPU.

The local language model is hosted through Ollama on the Legion. The final user interface will initially be Telegram, with WhatsApp considered later after the Telegram implementation is working reliably.

The project is intentionally divided into small milestones so that each layer can be understood and tested independently before the full system is assembled.

---

## 2. Current Position in the Learning Plan

The broader learning plan has already covered:

- Hardware and environment inspection
- Ollama installation and setup
- Running a local model
- Verifying GPU inference
- Understanding model packages and model lifecycle
- Learning Ollama runtime controls
- Understanding Ollama as a local HTTP model server

The project is currently at **Checkpoint 9: Build a Minimal Client**.

The immediate objective is not yet to build a polished multi-user chatbot. The objective is to build and understand the application layer that communicates with the Ollama server.

---

## 3. Final Vision

The eventual system should allow a user to send a message through Telegram and receive a response generated by a model running locally on the Legion GPU.

High-level data flow:

```text
Telegram user
    ↓
Python bot application running on the Legion
    ↓
Ollama HTTP API on localhost:11434
    ↓
Local model running on the RTX 4060 GPU
    ↓
Python bot application
    ↓
Telegram user receives the response
```

The Telegram bot is not the model server.

The Python application acts as the bridge between Telegram and Ollama.

Ollama remains responsible for:

- Loading and hosting the model
- Managing CPU, RAM, VRAM, and GPU execution
- Accepting HTTP API requests
- Running inference
- Returning generated responses

The Python application remains responsible for:

- Receiving messages from the user interface
- Constructing Ollama API requests
- Sending requests to the Ollama server
- Receiving and parsing Ollama responses
- Returning responses to the user interface
- Later managing context, configuration, access control, and persistence

## 4. Major Project Phases

The chatbot project is divided into two large functional phases.

## Phase 1 — One Chat

Build a working system for one user and one conversation.

The first version should assume:

- One Telegram chat
- One active conversation history
- One local Ollama server
- One selected model
- One Python bot application

The goal is to understand the complete message path before introducing multiple users, multiple histories, or a database.

Conceptual structure:

```text
Telegram
    ↕
Python bot application
    ↕
Ollama server
```

## Phase 2 — Multiple Chats

Extend the system so that multiple Telegram chats or users can use the bot independently.

Each Telegram `chat_id` must be associated with its own conversation history.

Conceptual structure:

```text
chat_id 123 → conversation history A
chat_id 456 → conversation history B
chat_id 789 → conversation history C
```

Phase 2 is divided further:

### Phase 2A — Multiple Chats in Memory

Store several chat histories in a Python dictionary or similar in-memory structure.

Example:

```python
histories = {
    123: [...],
    456: [...],
}
```

Characteristics:

- Multiple users can have separate contexts
- Histories exist only while the Python process is running
- Restarting the script deletes all histories
- No database is required yet

### Phase 2B — Persistent Multiple Chats

Store conversation histories outside the running Python process.

Possible storage options:

- JSON files
- SQLite
- A larger database system, if later justified

Characteristics:

- Histories survive application restarts
- Conversations can be restored
- Data ownership, schema, cleanup, and storage limits must be considered

Important distinction:

```text
Phase 2A = multiple chats with temporary storage
Phase 2B = multiple chats with persistent storage
```

A database is therefore not automatically required for the first multi-chat milestone.

## 5. Phase 1 System Architecture

Phase 1 has three major components.

## Component 1 — Telegram

Telegram is the eventual user interface.

Its responsibilities are limited to:

- Accepting user messages
- Delivering those messages to the Python bot application
- Displaying replies sent by the Python bot

Telegram does not run the model and does not directly communicate with the GPU.

## Component 2 — Python Bot Application

The Python application is the bridge and application logic layer.

It has four primary responsibilities:

1. Receive a Telegram message
2. Convert the message into an Ollama API request and send it
3. Receive and extract the model response
4. Send the response back through Telegram

Conceptually:

```text
Receive user message
    ↓
Build API request
    ↓
Send request to Ollama
    ↓
Receive Ollama response
    ↓
Extract generated text
    ↓
Return generated text to user
```

Although these responsibilities may later be separated into multiple functions or files, they remain part of one Python bot application.

## Component 3 — Ollama Server

The Ollama server runs locally on the Legion and exposes its API, normally at:

```text
http://localhost:11434
```

The server should be running whenever the bot is expected to answer messages.

The Ollama server:

- Owns the model process
- Loads the model into available memory
- Uses the RTX 4060 GPU for inference where supported
- Accepts JSON requests from the Python application
- Returns generated output

For the first local implementation, the Python application and Ollama server run on the same Legion computer. Therefore, the Python application can communicate with Ollama through `localhost`.

## 6. Phase 1 Plan of Action

Phase 1 will be developed in three major steps.

## Step 1 — Terminal Prototype

Build the first draft of the Python application without Telegram.

Instead of receiving and returning messages through Telegram, the first version uses the terminal.

Structure:

```text
Terminal input
    ↓
Python application
    ↓
Ollama API request
    ↓
Ollama server
    ↓
Model response
    ↓
Python application extracts response
    ↓
Terminal output
```

Initial behavior:

```text
You: Explain GPU memory bandwidth.
Model: ...
```

The first terminal prototype should prove that the Ollama-facing half of the system works.

Minimum responsibilities:

- Read a prompt from terminal input
- Construct a valid Ollama API request
- Send the request to the local Ollama server
- Receive the response
- Extract the generated text
- Print it to the terminal
- Handle basic connection or malformed-response errors

The terminal prototype should be designed so that the Ollama communication logic can be reused later.

Recommended internal structure:

```python
def ask_ollama(...):
    ...
```

The terminal input/output layer should call the reusable Ollama function rather than containing all HTTP logic directly.

This avoids rewriting the Ollama code when Telegram is added.

### Step 1A — Minimal One-Prompt Client

The smallest working milestone:

```text
Start script
→ enter one prompt
→ receive one model response
→ script exits
```

This verifies basic API communication.

### Step 1B — Interactive Terminal Chat

Extend the script into a loop:

```text
You: Hello
Model: Hello! How can I help?

You: What did I just say?
Model: You said "Hello."
```

This introduces conversation context.

The client application must retain and resend relevant message history. Separate API calls do not automatically become one continuous conversation merely because they use the same model.

Typical message structure:

```python
messages = [
    {"role": "user", "content": "Hello"},
    {"role": "assistant", "content": "Hello! How can I help?"},
    {"role": "user", "content": "What did I just say?"},
]
```

At this point, context may be stored only in memory.

### Step 1C — Reusable Internal Separation

Before Telegram integration, separate at least these concerns:

```text
Terminal interface
    ↓
Conversation/application logic
    ↓
Ollama API client
```

The project does not need excessive abstraction, but Telegram-specific code should not be mixed into the low-level Ollama request code.

---

## Step 2 — Telegram Integration

After the terminal prototype works, learn and implement Telegram communication.

The first Telegram milestone should not immediately involve Ollama.

### Step 2A — Telegram Echo Bot

Create a Telegram bot that receives a message and returns the same message or a fixed reply.

Example:

```text
User: hello
Bot: You said: hello
```

This isolates Telegram-specific concepts:

- Bot creation and token
- Receiving Telegram updates
- Message handlers
- Telegram chat IDs
- Sending replies
- Bot process lifecycle

Ollama should not be involved in this test.

### Step 2B — Connect Telegram to the Existing Ollama Client

Once the echo bot works, replace the echo logic with a call to the existing Ollama client function.

Structure:

```text
Telegram message
    ↓
Telegram handler
    ↓
Existing ask_ollama function
    ↓
Ollama server
    ↓
Model response
    ↓
Telegram handler
    ↓
Telegram reply
```

The Ollama API logic should be reused from Step 1, not rewritten.

For Phase 1, only one Telegram chat needs to be supported.

The bot only needs to work while:

- The Legion is powered on
- Ollama is running
- The Python bot application is running
- The Legion has internet access to communicate with Telegram

This is acceptable for the learning phase. Public hosting and 24/7 availability are not current requirements.

## Step 3 — Patching, Reliability, and Conversation Features

After the basic Telegram-to-Ollama path works, improve the single-chat system.

Possible upgrades include:

### Conversation Context

- Store user and assistant messages
- Resend the required history with each request
- Limit history to avoid uncontrolled context growth
- Add a `/reset` command to clear the current conversation

### Model and Runtime Configuration

- Select the model name from configuration
- Configure temperature, context length, and output limits
- Configure Ollama keep-alive behavior
- Avoid hard-coding settings throughout the source code

### Reliability

- Handle Ollama server not running
- Handle network failures
- Handle Telegram API failures
- Add request timeouts
- Handle empty or malformed responses
- Prevent the application from crashing on one failed message

### Output Handling

- Handle Telegram message-length limits
- Split long model replies when required
- Optionally support streamed output later
- Provide a temporary status such as "generating" only if useful

### Access Control

- Restrict the bot to the intended Telegram account or chat ID
- Prevent unknown users from consuming local GPU resources
- Keep the Telegram bot token outside the source code

### Logging

- Record errors
- Record request timing
- Record model name and relevant runtime settings
- Avoid logging sensitive conversation content by default unless intentionally enabled

## 7. Terminology

The following terms should remain distinct.

## Context

The messages supplied to the model so that it can respond consistently within the current conversation.

Example:

```text
Previous user messages
Previous assistant responses
Current user message
```

Context may exist only in memory.

## Persistence

Saving conversation data outside the running Python process so that it survives restarts.

Examples:

- JSON
- SQLite
- Database storage

## Long-Term Memory

A more advanced feature in which the application extracts, stores, and later retrieves selected facts or preferences rather than resending a raw transcript.

Long-term memory is not required for Phase 1 and should not be confused with ordinary conversation context.

## 8. Design Principles

The following principles should guide implementation.

### Build One Layer at a Time

Test terminal-to-Ollama before adding Telegram.

Test Telegram message handling before connecting it to Ollama.

### Reuse the Ollama Client Logic

The same Ollama request function used by the terminal client should later be called by the Telegram handler.

### Avoid Premature Databases

A database is not needed for the first single-chat implementation.

It is also not required for the first multi-chat implementation if histories are temporarily stored in a dictionary.

### Keep Phase 1 Single-Chat

Do not add multi-user storage, account systems, complex memory, or production deployment before the single-chat message path works reliably.

### Understand Every Boundary

The project should make the following boundaries clear:

```text
User interface
Application logic
HTTP API client
Model server
Inference runtime
GPU hardware
```

### Prefer Small, Verifiable Milestones

Each milestone should produce visible working behavior before the next layer is added.

## 10. Phase 1 Completion Criteria

Phase 1 is complete when all of the following are true:

- Ollama runs locally on the Legion
- The selected model performs inference on the local hardware
- A Python application can call the Ollama API
- The same Ollama client logic is reused by the terminal and Telegram interfaces
- A Telegram message can reach the Python application
- The Python application can send the message to Ollama
- The generated response can be returned to Telegram
- One conversation can retain context during the running session
- The conversation can be reset
- Basic connection and response errors are handled
- Unauthorized Telegram users can be rejected or ignored

Persistent storage and multiple independent chats are explicitly deferred to Phase 2.
